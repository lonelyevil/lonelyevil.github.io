    Docker镜像：第一部分 降低镜像大小

    /bin/cat
    2020-02-19
    翻译
    docker

    本文原作者为 Jérôme Petazzoni，发布于Ardanlabs Blog。
    如有错误或建议欢迎指出，谢谢。

    简介

    刚上手容器的时候，很容易就会被构建的镜像大小所吓到。在不妨碍到开发和运维的前提下，本系列将回顾一系列降低镜像大小的技术。

    在第一部分，首先会讨论多阶段构建，因为这是任何想降低镜像大小的人应当开始的地方。然后会解释静态和动态链接的区别和了解其中区别的原因。最后引出 Alpine。

    在第二部分，会讨论各种流行语言的特性。不仅是 Go，还会包括 Java、Node、Python、Ruby 和 Rust。除此以外，还会讨论更多全面利用 Alpine 的方法。

    在第三部分，会涉猎像共用基础镜像、剥离二进制和减少资源大小等有关于多数语言和框架的模式和反模式。接着会使用上一些例如 Bazel、Distroless、DockerSlim 或者 UPX 的外来高级技术。其中一些技术在某些场景会变成反生产而在其他特定场景可能非常有用。

    代码样例和所有提到的Dockerfile都方便地提供于一个公开的 Github 仓库，并且有一个Compose文件来构建所有镜像并轻松地比较大小。

    警告：使用经典镜像

    强烈建议给运行阶段使用经典镜像。经典是指一些像 CentOS、Debian、Fedora 和 Ubuntu 等熟悉的镜像。可能你已经听说了 Alpine 并且急切地想使用它。不要这样做！至少不是现在。之后会讨论 Alpine，然后会解释为什么要小心它。

    使用FROM scratch

    回到Hello World程序。C 版本大小是 16kB，Go 版本大小是 2MB。能得到这么小的镜像吗？能构建只包括二进制文件的镜像吗？

    但是在使用scratch作为基础镜像时，有一些东西必须牢记于心。

    没有 Shell

    scratch镜像没有 shell。这意味着不能在CMD或者RUN中用字符串语法。考虑下面的 Dockerfile。

    如果尝试docker run得到的镜像，会得到下列的错误信息

    错误信息展示得不是很清楚，但是核心信息在这：/bin/sh从镜像里消失了。因为在CMD或者RUN中用字符串语法，参数被传递给/bin/sh。这意味着CMD ./hello实际上执行了/bin/sh -c "./hello"，由于在scratch镜像中没有/bin/sh，故报错了。

    解决方法很简单：在 Dockerfile 里使用JSON 语法。CMD ./hello变成CMD ["./hello"]。在 Docker 检测到 JSON 语法时，它会直接执行而非通过 shell。

    没有调试工具

    根据定义，scratch是空的。它不带任何可以协助给容器除错的工具。不仅像上面说的那样没有 shell，还没有ls、ps、ping等等。这意味着不能进入容器（用docker exec或者kubectl exec）来调查。

    严格意义上，依旧有一些方法来给容器除错。docker cp可以把文件复制出容器；docker run --net container可以与网络栈交互；低阶工具nsenter可能非常强大。最近版本的 K8S 有暂时容器的概念，但是这依旧在 alpha 阶段。牢记这些技术肯定会使生活更复杂，尤其是已经很多该处理的。

    一种解决方法是用像busybox或者alpine来代替scratch。虽然这两者更大（busybox有 1.2MB 而alpine有 5.5MB），但是相较于动辄数百 MB 乃至上 G 巨大的整个应用的原始镜像来说，这个代价并不大。

    没有libc

    排除这个错误就更加看技巧。我们简单的Go语言写的hello world工作正常。但是如果尝试把C程序或者更复杂的比如使用了网络包的Go程序放到scratch镜像下，会得到以下的错误信息。

    一些文件看上去丢失了。但是报错信息并没有说明到底是哪个文件丢失了。丢失的文件是一个程序运行必须的动态库。什么是动态库？为什么需要它？在程序编译后，它和所需的库链接了起来。（小如上面的hello world程序都用到了库，函数puts就来自于一个库。）很久之前（在上个世纪九十年代之前），程序通常使用静态链接，意味着所有使用到的库都被包含在二进制文件中。但是在一个像 Linux 一样的时分系统中，我们同时运行许多储存于硬盘的程序，这些程序几乎一直使用标准 C 库。在这种场景下，使用动态链接更有优势。用了动态链接，最终的二进制文件中就不需要包含所有用到的库。取而代之，包含这些库的引用，就像“这个程序需要从libtrigonometry.so调用函数cos、sin和tan。在程序执行时，系统搜寻libtrigonometry.so并和程序一起加载，从而程序能够调用这些函数”。

    动态链接有不少优势。

    1. 它节省了硬盘空间，共同的库再也不需要被重复。
    2. 它节省了内存，这些库可以从硬盘载入一次，然后在不同使用库的程序间共享。
    3. 它让维护更加容易，因为当库升级时，不需要重编译所有依赖库的程序。

    仔细说来，内存节省并不是动态库导致的而是共享库。话虽如此，这两个往往是一种东西。还记得在 Linux 中，动态库文件通常以.so为扩展名吗？这里的so意为共享对象（shared object）。在 Windows 里，扩展名是.dll，表示动态链接库。

    回到主题，默认C程序是动态链接的。对于用了某些包的Go程序来说也是如此。文中这个特定的程序用到了标准 C 库，在最近的 Linux 系统中应为文件libc.so.6。所以要运行程序，在镜像中程序需要这个文件存在。如果使用scratch，这个文件明显是没有的。使用busybox或者alpine也一样，busybox不带标准库而alpine使用另一个不兼容的标准库。后面会讨论更多。

    如何解决这个问题呢？这里有三种选项

    静态构建二进制文件

    在编译时告诉工具链构建静态二进制文件。取决于如何构建程序，达成这个目标的方法多种多样。例如这里使用gcc，所有需要做的就是在命令行参数中添加-static开关。

    最终的二进制文件从 16kB 增长到了 760kB（在作者系统上）。当然，因为嵌入了库，所以文件变大了。但是这个二进制文件就可以正确地在scratch镜像中运行了。

    在上述例子中，真正的库只有libc.so.6。linux-vdso.so.1和一个叫做 VDSO 加速系统调用的技术有关。假装它不在这。至于ld-linux-x86-64.so.2，这就是动态链接器本身。技术上而言，hello二进制文件包含了这样的信息：“嘿，这是一个动态链接程序，ld-linux-x86-64.so.2知道怎么把它的所有部分组合在一起”。

    如果想这样做的话，可以手动把ldd列出来的文件都加到镜像中。过程非常冗长，而且难以维护，尤其是依赖了很多库的程序。对于小小的hello world程序来说这种方法可行。但是对于更复杂的程序，例如一个程序用到了 DNS，就会碰到其他问题。系统中常用的 GNU C 库利用了相当复杂的叫做名称服务开关（Name Service Switch）的技术实现了 DNS 等几个功能。这个技术需要一个配置文件/etc/nsswitch.conf和另外的库。但是这些文件不显示在ldd的记录中，因为它们是在程序运行的时候再加载的。如果需要在容器中解析 DNS，依旧需要包含它们（它们的位置一般在/lib64/libnss_*）。

    笔者个人不建议这种方法，这个方法太晦涩，难以维护，将来还可能破坏。

    使用busybox:glibc

    有个镜像设计来解决所有这种问题，叫做busybox:glibc。这是一个利用busybox来提供很多好用的除错和操作工具的小镜像（5MB）。它还包含了 GNU C 库（glibc）。这个镜像正好包含了之前提到的所有麻烦的文件。这是如果想在一个小镜像中运行一个用了动态库的二进制文件所需要的镜像。

    但是记住，如果程序用到了额外的库，这些库也需要另外复制进来。

    这是 12000 倍的大小减少，换句话说是少了 99.99%硬盘占用。不错。笔者个人而言不会使用scratch镜像，因为除错会很麻烦。但是如果这就是读者追求的，那它就在这。
